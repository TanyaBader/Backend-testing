## Задание 3. Базы данных

#### База данных (БД) - это хранилище данных, которое может быть структурировано разными способами.

#### Система управления базами данных (СУБД) - это программное обеспечение, которое помогает управлять и манипулировать данными в базе данных

Типы БД:

- Иерархические БД организованы как дерево, где связи устанавливаются в виде "родитель-потомок".
- Сетевые БД имеют более сложную структуру с множественными связями между записями.
- Реляционные БД (SQL) основаны на таблицах с данными, связанными через общие ключи.
- Нереляционные БД  (известные как NoSQL) структурируют данные не по табличной схеме, а используют форматы, такие как ключ-значение, документы, колонки или графы.

Типы СУБД:

Реляционные СУБД:

- MySQL: Открытая система, широко используемая в веб-разработке. На курсе мы будем работать с ней.
- Oracle Database: Коммерческая СУБД, широко используемая в корпоративных решениях.
- Microsoft SQL Server: Предназначена для интеграции с другими продуктами Microsoft.
- PostgreSQL: Открытая СУБД, поддерживает большие базы данных и сложные запросы.

Нереляционные (NoSQL) СУБД:

- MongoDB: Документо-ориентированная СУБД, хранит данные в формате похожем на JSON.
- Cassandra: Ориентирована на работу с большими объемами данных.
- Redis: СУБД на основе ключ-значение, используется для кэширования данных и реализации очередей сообщений.

Реляционная БД хранит данные в виде таблиц:

- строки - записи,
- столбцы - атрибуты,
- связи - через ключи (PRIMARY / FOREIGN KEY).
- Используется SQL и строгая схема данных.
    - Ключевые особенности:
        - заранее заданная структура (schema);
        - связи между таблицами (JOIN);
        - транзакции (ACID);
        - строгая целостность данных.
    - Плюсы реляционных БД:
        - Целостность данных(невозможно создать «битые» связи);
        - Транзакционность (ACID)подходит для денег, аккаунтов, заказов;
        - Мощные запросы JOIN, GROUP BY, агрегаты;
        - Хорошая предсказуемость и понятное поведение системы.
    - Минусы реляционных БД:
        - Плохо масштабируются горизонтально(шардинг сложен);
        - Жёсткая схема(сложно часто менять структуру);
        - Медленнее при огромных объёмах write/read(миллиарды событий).
    - Где применяются:
        - банковские системы;
        - ERP / CRM;
        - учёт, биллинг;
        - пользовательские профили.

Нереляционные БД:

- не используют таблицы и JOIN’ы,
- ориентированы на масштабирование и скорость,
- допускают гибкую структуру данных.
    - Плюсы NoSQL:
        - Горизонтальное масштабирование(легко распределять по серверам);
        - Высокая скорость(особенно на запись);
        - Гибкая схема(можно хранить разные структуры);
        - Отказоустойчивость(репликация «из коробки»).
    - Минусы NoSQL:
        - Слабая или отсутствующая целостность(часто eventual consistency);
        - Нет сложных JOIN’ов(логику приходится переносить в код);
        - Разные стандарты(нет единого «SQL для NoSQL»).
    - Где применяются:
        - мессенджеры;
        - социальные сети;
        - аналитика, логи;
        - real-time сервисы.

### Гипотеза:

Выбранный сервис: Альфа-банк

По моему мнению, для Альфа-банка основным типом баз данных является реляционная. В ней используется строгая схема данных и применяется в банковской системе. Она поддерживает транзакционность, что подходит для перевода денежных средств. Жесткая схема структуры, целостность данных и хорошая предсказуемость также подходит банковской структуре. Я считаю именно это позволяет обезопасить клиента и его денежные средства.
Но также могут присутствовать и не реляционные базы данных, так как они применяются в real-time сервисах, что необходимо для быстрого обновления данных и минимальной задержки.

    - Тип БД:
        - Нереляционная БД (Высокая скорость.)

### Тест-кейсы для проверки выбранных типов БД.

1. **Проверить, что денежный перевод выполняется и данные не теряются.**

Предусловие: Пользователь авторизован в системе и у него есть два счёта с положительным балансом.

- Шаги:
    - Инициировать перевод между счетами;
    - Во время операции смоделировать ошибку (например, обрыв соединения);
    - Проверить балансы обоих счетов после ошибки.
- Ожидаемый результат:
    - Деньги либо не списались вообще, либо корректно списались и зачислились;
    - Не возникает частичного списания.

2. **Проверить, что при некорректных данных операция не выполняется.**

Предусловие: Пользователь авторизован в системе, система доступна и содержит корректные данные.

- Шаги:
    - Попробовать выполнить операцию по счёту, который не существует;
    - Попробовать создать транзакцию без привязанного пользователя.
- Ожидаемый результат:
    - Операция отклоняется;
    - Система возвращает ошибку.

3. **Проверить, что уведомления и статусы обновляются сразу.**

Предусловие: Пользователь авторизован в системе, включены уведомления, приложение активно и имеет стабильное интернет-соединение.

- Шаги:
    - Выполнить платёж или перевод;
    - Открыть мобильное приложение;
    - Отследить появление push-уведомления и изменение статуса операции.
- Ожидаемый результат:
    - Уведомление приходит практически сразу;
    - Статус операции обновляется без перезагрузки.
